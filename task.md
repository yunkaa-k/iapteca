**Теорія**

**Термінальні CLI-агенти (як-от Claude Code, Kiro, Aider) та інтелектуальні IDE (на кшталт Cursor, Windsurf, Trae, Void) мають кардинально різні зони ефективності в IT.**

**Нижче наведено перелік типових завдань, з якими ці інструменти справляються найкраще, розділений за їхньою архітектурною спеціалізацією.**

**1. Конфігурування інфраструктури та IaC (Infrastructure as Code)**

**Для хмарних інженерів, системних адміністраторів та DevOps-фахівців термінальні агенти стали повноцінними інструментами автоматизації.**

**Написання та рефакторинг маніфестів (Terraform, OpenTofu, Ansible): Агенти можуть з нуля згенерувати структуру модулів для розгортання мережевої інфраструктури (наприклад, AWS VPC, Transit Gateway, налаштування маршрутизації та Security Groups) відповідно до кращих практик безпеки.**

**Діагностика та аналіз логів (Troubleshooting): Ви можете згодувати CLI-агенту сирий лог помилки з пайплайну CI/CD або вивід команди journalctl / docker logs. Агент аналізує помилку конфігурації, знаходить проблемний файл у репозиторії та самостійно виправляє його (наприклад, виправляє синтаксис в Nginx config або SELinux політиках).**

**Автоматизація рутинних скриптів: Створення Bash, Python або Go скриптів для парсингу API, автоматичного бекапу, очищення застарілих хмарних ресурсів або збору метрик.**

**2. Багатофайлова розробка та архітектурний рефакторинг**

**У цих завданнях лідирують AI IDE завдяки графічному інтерфейсу та технологіям глибокого аналізу контексту проєкту.**

**Впровадження наскрізних змін (Cross-file refactoring): Завдання на кшталт "Перепиши логіку автентифікації користувача з сесій на JWT". Інструменти на зразок Cursor (Composer) або Windsurf (Cascade) самостійно знайдуть усі пов'язані файли (контролери, моделі, проміжне ПЗ), внесуть туди зміни та підготують акуратний Diff для перевірки.**

**Онбординг та аналіз чужого коду (Legacy Code): Коли ви відкриваєте великий незнайомий проєкт, ви можете попросити IDE побудувати архітектурну мапу: "Поясни, як у цьому проєкті влаштована маршрутизація запитів і де оголошені інтерфейси для роботи з базою даних".**

**Генерація Unit-тестів: Швидке покриття коду тестами (PyTest, Jest, Go testing). Графічні IDE аналізують усі граничні випадки функцій і генерують моки (mocks) та тестові сценарії за один клік.**

**3. Повсякденна розробка та Git-workflow**

**Ці завдання виконуються однаково ефективно в обох типах інструментів, але кожен має свій підхід.**

**Автоматичний Git-менеджмент (фішка Aider): Ви даєте текстову команду "Додай валідацію полів у форму реєстрації". Агент робить зміни, самостійно запускає лінтер, перевіряє, що код чистий, і робить Git-коміт із чітким описом на кшталт feat: add validation rules for registration form.**

**Міграція між технологіями: Перенесення коду зі старіших версій мов на нові (наприклад, міграція з Python 3.8 на 3.11) або порт коду (наприклад, переписати скрипт автоматизації з Bash на Go для кросплатформеності).**

**Інтеграція з базами даних: Створення складних SQL-запитів, написання файлів міграцій (Liquibase, Flyway, Sequelize) та оптимізація існуючих повільних запитів через аналіз EXPLAIN ANALYZE.**

**Підсумок: Що під яке завдання обрати?**

**Якщо ваше завдання інфраструктурне або термінальне (покопатися в Linux, написати Terraform-модуль, розпарсити лог шлюзу, швидко пофіксити баг через SSH) ➡️ запускайте Aider або Claude Code.**

**Якщо ваше завдання продуктове та архітектурне (написати новий мікросервіс, зробити рефакторинг API, покрити тестами фронтенд) ➡️ відкривайте Windsurf, Cursor або безкоштовний Trae.**

**Розподіл інструментів: Студент має обрати для порівняння:**

**один інструмент з Групи А (Cursor, Windsurf, Trae, PearAI, Void) та**

**один інструмент з Групи Б (Claude Code, Kiro, Aider, Antigravity).**

**Порядок виконання роботи**

**Крок 1: Підготовка Baseline-інфраструктури:Git-ізоляція та налаштування оточення.**

**Студент робить форк тестового репозиторію магазину. Для чистоти експерименту створюються дві окремі гілки: feature/ai-ide-test (для Групи А) та feature/ai-cli-test (для Групи Б).**

**Забороняється вносити зміни ручним написанням коду — всі модифікації мають виконуватись виключно через промпти до обраних ШІ-інструментів.**

**Крок 2: Багатофайловий рефакторинг бізнес-логіки:Тестування AI IDE (Група А).**

**Завдання для графічних ШІ-середовищ (наприклад, Trae або Windsurf).**

**Промпт-задача: "Проаналізуй сервіс Order-Processing та User-Auth. Додай нову фічу 'Промокоди'. Створи новий мікросервіс Promo-Service, налаштуй gRPC або REST комунікацію між ним та сервісом замовлень. Онови базу даних (додай таблицю promo\_codes) та згенеруй Unit-тести для перевірки розрахунку знижки."**

**Критерій успіху: Агент має самостійно зв'язати кілька файлів, не зламавши існуючу логіку авторизації.**

**Крок 3: Infrastructure as Code (Terraform) та AWS:Тестування CLI Агентів (Група Б).**

**Завдання для термінальних агентів (наприклад, Aider або Claude Code).**

**Промпт-задача: "Напиши Terraform-модулі для розгортання цієї мікросервісної архітектури в AWS. Створи VPC з публічними та приватними підмережами. Налаштуй ECS (Fargate) кластер для запуску Docker-контейнерів сервісів. Забезпеч Least Privilege доступ: створи IAM ролі так, щоб Product-Catalog мав доступ лише на читання до бази даних, а Order-Processing — на запис."**

**Критерій успіху: Згенерований код має проходити перевірку terraform plan без синтаксичних помилок.**

**Крок 4: Security-аудит та фікс CI/CD пайплайну:Перехресне тестування автономності обох груп.**

**Завдання для обох обраних інструментів. У репозиторій заздалегідь додано GitHub Actions пайплайн із "вшитим" AWS секретним ключем (Hardcoded Secret) та помилкою у версіонуванні Docker-образу (тег :latest замість хешу коміту).**

**Промпт-задача: "Проведи аудит безпеки CI/CD пайплайну. Знайди вразливості, винеси секрети в Environment Variables (або AWS Secrets Manager) та виправ логіку версіонування."**

**Критерій успіху для CLI: Здатність самостійно зробити git commit із правильним описом виправлення безпеки (наприклад, в Aider).**
